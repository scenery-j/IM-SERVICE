# Описаниесистемы
### 1. Проект предоставляет внешним системам мгновенные коммуникационные услуги (личное общение и групповые чаты) и одновременно позволяет реализовать пользовательскую синхронизацию сообщений между несколькими клиентами (web, ios, android), кэширование сообщений при offline, изменение онлайн‑статуса, интеграцию с множеством систем, а также интеллектуальный диалог (интеграция с большим языковым моделью Tongyi Qianwen) и другие функции.
#### 1. Бизнес‑система: запуск Netty‑сервиса (реализация пользовательского протокола) для отправки сообщений в личных и групповых чатах, а также других бизнес‑логик.
#### 2. Внешние бизнес‑системы: реализация трёх бизнес‑систем входа и пользовательских функций.
#### 3. Фронтенд‑сервис: реализация взаимодействия пользователей в системах мгновенного обмена сообщениями
### 2. Технологический стек
#### Netty: запуск netty‑сервера, реализация tcp‑соединения и пользовательского протокола для реального времени между клиентом и сервером.
#### dubbo: реализация tcp‑микросервисов и вызова методов между бизнес‑микросервисами
#### rabbitmq: распределение сообщений
#### mysql: хранение бизнес‑данных#### redis: хранение отложенных сообщений и горячих данных
#### naocs: реализация сервисного обнаружения и регистрации для tcp‑микросервисов, а также регистрации dubbo
#### Spring Boot: микрофреймворк
### 3. Описание модулей backend‑сервисов
#### im-codec: реализация, связанная с пользовательским протоколом Netty
#### im-common: общие_entity и утилитные классы
#### im-message-store: микросервис хранения сообщений
#### im-service: бизнес‑микросервис IM‑сервиса — реализация бизнес‑логики
#### im-tcp: модуль netty‑сервиса — реализация соединения с клиентом и обмена сообщениями### 4. Оптимизация бизнес‑процессов
#### Мгновенность сообщений: как обеспечить своевременный отклик отправителя получателю, основной подход — минимизация функций от отправки до получения сообщения: 1) использование пула потоков для параллелизма (вызывает проблемы с порядком сообщений) 2) асинхронное хранение сообщений, чтобы избежать длительных операций записи в базу 3) проверка законности отправки перенесена на уровень tcp, использование удалённого вызова сервисов через dubbo для избежания нелегальных запросов, занимающих ресурсы
#### Порядокность сообщений: из‑за использования потоков и асинхронного хранения порядок сообщений не гарантируется. Для решения вводится поле sequence, которое получает значение через механизм автоувеличения в redis.
#### Надёжность сообщений: чтобы гарантировать доставку сообщения получателю: 1) добавление ack — подтверждение прибытия сообщения на сервер и его доставки получателю 2) клиентская механика отката, при которой при отсутствии ack сообщение повторяется (вызывает проблемы с идемпотентностью)
