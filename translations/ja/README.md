# システムアーキテクチャの紹介
### 一、本プロジェクトは外部に提供するリアルタイム通信サービス（単 chat とグループチャット）に加え、カスタムマルチ端末（web、iOS、android）メッセージ同期、オフラインメッセージキャッシュ、オンラインステータス変更、複数システムの接続、インテリジェントな対話（通義千問大モデルとの統合）などの機能を実現できます。
#### 1、業務システム：Netty サービスを起動（カスタムプロトコルの実装）により単 chat / グループチャットのメッセージ送信、リアルタイム通信業務の他の業務ロジックを実装。
#### 2、サードパーティ業務システム：3つの業務システムのログインとカスタムの業務機能の実装。
#### 3、フロントサービス：リアルタイム通信のユーザーインタラクションの実装
### 二、技術スタック
#### Netty：Netty サーバーを起動し、TCP 接続を確立し、カスタムプロトコルでクライアントとサーバーのリアルタイム通信を実現。
#### dubbo：TCP マイクロサービスと業務マイクロサービス間のインターフェース呼び出しを実現。
#### rabbitmq：メッセージの配信を実現。
#### mysql：業務データの永続化。
#### redis：オフラインメッセージの永続化とホットデータのキャッシュ。
#### nacos：TCP マイクロサービスのサービス登録と発見、dubbo レジストリセンター。
#### Spring Boot：マイクロサービスフレームワーク。
### 三、バックエンドサービスモジュールの紹介
#### im-codec：Netty カスタムプロトコルに関する実装。
#### im-common：共通エンティティクラスとユーティリティクラス。
#### im-message-store：メッセージ永続化マイクロサービス。
#### im-service：IM サービス業務マイクロサービス-業務ロジックの実装。
#### im-tcp：Netty サービス-クライアントとの接続・メッセージ交換モジュール。
### 四、ビジネス最適化#### メッセージの即時性：送信側のメッセージが受信側に迅速に応答することを保証する方法。基本的なアプローチは、メッセージ送信から受信までのビジネスロジックを単一の行に分割すること：1、スレッドプールを使用して並列処理（メッセージの順序性問題を引き起こす可能性） 2、メッセージストレージを非同期で実装し、データベースへの書き込みの遅延を回避。 3、メッセージ送信の合法性検証を TCP レイヤに移し、Dubbo リモート呼び出しの service レイヤのビジネスメソッドを利用して不要なリソース消費を防止。
#### メッセージの順序性：メッセージ送信にスレッド並列と非同期ストレージを使用すると順序が保証できない。そのためシーケンスフィールドを導入し、Redis の自動インクリメント機構で seq 値を取得する実装。
#### メッセージの信頼性：メッセージが必ず受信側に届くようにするため：1、ACK を導入し、サーバーにメッセージが到達し、受信側に到達したことをクライアントに通知。2、クライアントは ACK を受信しない場合に再送機構を提供し、重複処理（イデム性問題）に対応。
#### メッセージのイデム性：メッセージの再送が存在するため、クライアントは一度だけメッセージを受け取る必要がある。サービスは Redis に既に処理済みのメッセージキーを保存し、超時設定を設定する。キーが存在すれば既に処理済みとしてスキップし、重複処理を防止。

# IM-SERVICE 即時通讯系统

基于 Spring Cloud、Netty 的分布式即时通讯系统。

## システムアーキテクチャ

システムは以下の主要サービスを含みます：

- `im-service`: 核心業務サービス
- `im-message-store`: メッセージストレージサービス
- `im-tcp`: TCP 長期接続サービス

## 環境要件

- JDK 8+
- Maven 3.6+
- Docker 20.10+
- Docker Compose 2.0+
